# Variáveis de ambiente

O MB CLI mescla variáveis de várias fontes e injeta o resultado no **processo filho** (plugins ou `mb run`). O ambiente do seu shell **não é alterado**.

## Resumo

| O quê | Onde configurar | Persistente? |
|-------|-----------------|--------------|
| Defaults globais | `env.defaults` em `~/.config/mb/` (ou equivalente no macOS) | ✅ |
| Por contexto (staging, prod…) | Vault com `--env-vault <nome>` → `.env.<nome>` no diretório de configuração | ✅ |
| Por projeto | Chave `envs` no `mbcli.yaml` da raiz do projeto | ✅ (versionado) |
| Por execução | `--env KEY=VALUE` | ❌ |

Para definir, listar e remover variáveis, use **`mb envs`**. [Referência completa →](../commands/envs.mdx)

## Ordem de precedência

Da **menor** para a **maior** precedência:

| # | Camada | Quando entra |
|---|--------|--------------|
| 1 | **Sistema** | Sempre — variáveis do shell (`os.Environ()`) |
| 2 | **`env.defaults`** | Sempre — ficheiro base no diretório de configuração (ex.: `~/.config/mb/env.defaults` no Linux) |
| 3 | **Vault (`--env-vault`)** | Quando se passa `--env-vault <nome>` → `.env.<nome>` no mesmo diretório |
| 4 | **`mbcli.yaml` → `envs`** | Sempre — escalares na raiz de `envs` (vault de projeto **default**). Com `--env-vault <nome>`, também o sub-mapa homónimo (`envs.<nome>`) |
| 5 | **`.env` no cwd** | Quando existe um `.env` no diretório de trabalho atual |
| 6 | **`--env-file`** | Quando se passa `--env-file <path>` |
| 7 | **`env_files` do manifest** | Só plugins — arquivos `.env` declarados no `manifest.yaml` para o vault efetivo |
| 8 | **`--env KEY=VALUE`** | Sempre — maior precedência, sobrescreve tudo |

> **Vaults de projeto (`project` / `project/<nome>`)** são lógicos — vêm do `mbcli.yaml` e não podem ser usados com `mb envs set --vault` nem `--env-vault`. Veja [Vaults de projeto](#project-vaults) abaixo.

### Exemplo de `envs` no `mbcli.yaml`

```yaml
envs:
  API_BASE: https://api.example.test
  FEATURE_X: "1"
  staging:
    API_BASE: https://api.staging.example
```

- `API_BASE` e `FEATURE_X` → vault de projeto **`project`** (sempre mesclados)
- `staging.API_BASE` → vault de projeto **`project/staging`** (só com `--env-vault staging`)

## Secrets

### Keyring do sistema (`--secret`)

O valor fica no keyring (macOS Keychain, Linux Secret Service), não em ficheiro. Na listagem aparece como `***`. Use `mb envs list --show-secrets` para ver.

### 1Password (`--secret-op`)

O valor fica no cofre 1Password; a referência `op://` é gravada em `*.opsecrets`. Requer `op` no PATH e sessão ativa.

```bash
mb envs set TOKEN=abc --secret-op
mb envs set TOKEN=abc --secret-op --vault staging  # cofre específico
```

Em CI, adicione `--yes` para evitar prompt.

## Vaults de projeto {/* #project-vaults */}

O ficheiro `mbcli.yaml` na raiz do projeto pode definir variáveis sob a chave `envs`:

- **Escalares na raiz** → vault lógico **`project`** (sempre mesclados)
- **Sub-mapas** → vaults lógicos **`project/<nome>`** (só com `--env-vault <nome>`)

Diferente dos vaults em disco (`.env.<nome>` no diretório de configuração):

- Não suportam secrets (sem keyring, sem `op://`) — só valores em claro
- São versionados no repositório
- O nome `project` e prefixo `project/` são reservados

Para listar vaults de projeto:

```bash
mb envs vaults              # mostra project, project/staging, etc.
mb envs list --vault project        # só a raiz de envs no YAML
mb envs list --vault project/staging # só o sub-mapa staging
```

## Quando usar o quê

| Cenário | Recomendação |
|---------|-------------|
| Variável usada em todos os projetos | `mb envs set` (env.defaults) |
| Variável por ambiente (staging, prod) | Vault: `mb envs set --vault staging` + `--env-vault staging` |
| Variável compartilhada pela equipe | `envs` no `mbcli.yaml` (versionado) |
| Secret (tokens, senhas) | `mb envs set --secret` (keyring) ou `--secret-op` (1Password) |
| Override rápido numa execução | `--env KEY=VALUE` |
| Arquivo `.env` de projeto | `.env` no cwd ou `--env-file` |

## `mb run`

Execute qualquer programa com o mesmo ambiente mesclado dos plugins:

```bash
mb run python script.py
mb run --env-vault staging -e FOO=bar uv sync
```

Flags podem ir antes ou depois de `run`, sempre antes do executável. Depois de `--`, nada é interpretado como flag do MB.

## Tema do gum

O CLI injeta variáveis `GUM_*` para scripts de plugins usarem cores consistentes (laranja `#FFA500` para cabeçalhos, verde `#00A86B` para seleção). Para sobrescrever, defina as mesmas chaves em `env.defaults` ou com `--env`.
