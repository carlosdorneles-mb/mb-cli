# `mb alias`

Comandos para criar **atalhos** (aliases) persistentes no MB CLI: você associa um **nome curto** a uma lista de argumentos de programa — o mesmo formato que o sistema usa para `argv`. O MB grava isso em disco, gera scripts de shell e pode acrescentar ao **perfil do seu shell** um bloco que carrega esses scripts automaticamente.

**Em resumo:**

- No **terminal**, depois de carregar o perfil, você pode digitar o nome do alias como se fosse um comando — ele roda com o ambiente **normal** daquele shell (sem o merge de variáveis do MB).
- Com **`mb run <nome>`**, o MB troca o nome pelos argumentos salvos e aplica o **mesmo ambiente mesclado** dos plugins (incluindo vault, se estiver configurado no alias).

Para variáveis globais, vaults e ordem de precedência, veja [Variáveis de ambiente](../user-guide/environment-variables.mdx). Para detalhes de execução, veja [`mb run`](./run.mdx).

## Onde os arquivos ficam (sem supor um único caminho)

Tudo abaixo fica no **diretório de configuração do MB** — a pasta onde também estão `config.yaml` e `env.defaults`. O caminho **depende do sistema operacional** (por exemplo Linux costuma seguir `XDG_CONFIG_HOME`; no macOS costuma ser outra pasta dentro de *Application Support*). Tabela resumida: [Instalação — Diretórios por sistema operacional](../getting-started/installation.mdx#diretórios-por-sistema-operacional).

- **`aliases.yaml`** — na raiz desse diretório. Esquema: `version` e, para cada alias, `command` (lista de strings) e opcionalmente `env_vault` (nome do vault usado só em `mb run`).
- **`mbcli.yaml` (repositório)** — opcional; ver seção abaixo. Aliases salvos aqui **não** geram funções em `shell/`; só funcionam com **`mb run`** (e aparecem no `mb alias list` com origem **repositório**).
- **Subpasta `shell/`** — scripts gerados (`aliases.bash`, `aliases.fish`, `aliases.ps1`, etc.) que o bloco do perfil passa a carregar (apenas a partir de `aliases.yaml`).

## Aliases em `mbcli.yaml` (repositório) {/* #aliases-em-mbcliyaml-repositório */}

O caminho do arquivo segue as mesmas regras que `envs` no repositório: `MBCLI_YAML_PATH`, ou `${MBCLI_PROJECT_ROOT}/mbcli.yaml`, ou `./mbcli.yaml` no diretório atual. Ver [Variáveis de ambiente — `mbcli.yaml`](../user-guide/environment-variables.mdx).

- **`mb alias set … --mbcli-yaml`** / **`mb alias unset … --mbcli-yaml`** — grava ou remove entradas na chave `aliases` desse arquivo. Se o arquivo não existir, o `set` cria um mínimo com `aliases`.
- **Precedência em `mb run`:** se o mesmo nome existir em `aliases.yaml` e em `mbcli.yaml`, vale a definição do **repositório** (`mbcli.yaml`).
- **Formato `aliases`:** espelha `envs`: cada chave pode ser um mapa com `command` (lista) e `env_vault` opcional **na raiz**, ou um **submapa** (grupo) cujo nome vira `env_vault` implícito para os aliases dentro dele, salvo se o próprio alias definir `env_vault` (use `env_vault: ""` para forçar sem vault dentro de um grupo). Os nomes de grupo **`project`** e prefixo **`project/`** são reservados.
- **Regravar o YAML** com o MB pode alterar a ordem das chaves e **não preserva comentários** no arquivo.

## Perfil do shell e flag `--shell`

Ao rodar qualquer subcomando de `mb alias` (exceto pedidos de ajuda ou `mb alias` sozinho, que só mostram ajuda), o MB garante no **arquivo de perfil** do shell um bloco delimitado por:

```text
# mb-cli user aliases BEGIN
# mb-cli user aliases END
```

Dentro do bloco há uma linha que carrega os scripts da subpasta `shell/` do diretório de configuração. O bloco é **idempotente**: rodar `mb alias` de novo não duplica o trecho.

- **`mb alias --shell <tipo>`** — força o shell alvo (`bash`, `zsh`, `fish`, `powershell`). Se você omitir, o MB usa a variável `SHELL` para escolher o perfil.
- Para **tirar só** a integração com o MB, edite o arquivo de perfil e apague (ou comente) tudo entre `BEGIN` e `END`.

Depois de mudanças, abra um terminal novo ou rode `source` no arquivo de perfil que o MB indicou.

## `mb alias set`

Cria um alias novo ou **atualiza** um que já existe (comando e/ou vault).

### Sintaxe

```text
mb alias set <nome> [--mbcli-yaml] [--env-vault <vault>] [--yes] [-- <comando>...]
```

- **`--`** — obrigatório **antes** do primeiro token do comando que você quer guardar. Tudo depois do `--` vira argumento do alias; o MB não trata como flag dele.
- **`--mbcli-yaml`** — grava no `mbcli.yaml` do repositório em vez de `aliases.yaml`; **não** regenera os scripts de shell.
- **`--env-vault <vault>`** — opcional. Se estiver definido, **`mb run <nome>`** usa esse vault no merge (arquivo `.env.<vault>`, secrets daquele vault, etc.). Chamar o alias **direto no shell** (sem `mb run`) **não** aplica esse merge.
- **`--yes` / `-y`** — confirma alteração em alias **já existente** sem perguntar no terminal (útil em CI ou scripts).

### Atualizar só o vault

Com **apenas** o `<nome>` na linha e a flag **`--env-vault`** presente (incluindo `""` para limpar), o MB só mexe no vault associado — o comando gravado continua o mesmo — desde que o alias já exista.

```bash
mb alias set dev --env-vault staging
mb alias set dev --env-vault ""
```

### Confirmação

Criar alias **novo** não pede confirmação. **Mudar** comando ou vault de um alias que já existe pede confirmação em terminal interativo; fora disso use `--yes`.

### Exemplos

```bash
mb alias set dev -- docker compose up

mb alias set build --env-vault staging -- npm run build

# Para o merge do mb run enxergar variáveis: use um shell com -c (o mb run não expande $ nos argumentos sozinho)
mb alias set api --env-vault staging -- sh -c 'echo API=$API_KEY'

# Quebra de linha no terminal (barra invertida antes de Enter)
mb alias set dev -- \
  docker compose \
  up -d

mb alias set dev --yes -- docker compose pull

# Alias versionado no repositório (só mb run; não entra no shell)
mb alias set ci --mbcli-yaml -- npm run ci
```

### Aspas, `$` e `mb run`

1. **Na hora de definir o alias**, o **seu** shell (bash/zsh) interpreta a linha primeiro. Aspas **duplas** fazem variáveis expandirem **antes** do MB receber os argumentos. Para guardar texto com `$` literal, prefira **aspas simples** no shell (por exemplo `'echo $HOME'`) ou escape correto.
2. O **`mb run`** chama o programa com **lista de argumentos direta** — não há shell no meio. Por isso `$VAR` nos argumentos **não** vira valor automaticamente. Para usar variáveis vindas do ambiente mesclado pelo MB, o comando do alias precisa passar por um **shell** que faça essa expansão, por exemplo: `sh -c 'echo aqui $MINHA_VAR'`.

## `mb alias list`

Lista todos os aliases em ordem alfabética pelo nome.

### Modos de saída

| Situação | O que aparece |
|----------|----------------|
| **Terminal** (stdout é TTY) | Tabela **fzf** com colunas **ALIAS**, **VAULT** e **FONTE** (config = `aliases.yaml`, repositório = `mbcli.yaml`) e painel de **preview** com o comando completo. Exige `fzf`, `gum` e `jq` no `PATH`. |
| **Pipe ou redirecionamento** | Tabela com **gum** (ALIAS, VAULT, COMANDO truncado, FONTE). |
| **`--json`** | Sempre imprime um array JSON (não depende de TTY), bom para usar com `jq`. |

### Formato `--json`

Cada item do array tem:

| Campo | Tipo | Significado |
|-------|------|-------------|
| `name` | string | Nome do alias |
| `envVault` | string | Vault associado (pode ser vazio) |
| `command` | array de strings | Argumentos do alias (`argv`) |
| `source` | string | `config` (arquivo global) ou `project` (`mbcli.yaml`) |
| `mbcliPath` | string | Caminho absoluto do `mbcli.yaml` quando `source` é `project`; vazio caso contrário |

```bash
mb alias list --json | jq '.[] | select(.source == "project")'
```

### Preview e o caractere `$`

No modo interativo, o preview mostra o comando **como está salvo** no arquivo; o texto com `$` não deve sumir por causa de expansão indevida no script de preview. O painel inclui também **Origem** (`config` ou `project`, como no JSON de `--json`) e, para aliases de repositório, o caminho absoluto do **`mbcli.yaml`**; na tabela fzf, a coluna **FONTE** usa os rótulos **config** e **repositório**.

## `mb alias unset`

Remove um ou mais aliases de uma vez.

### Sintaxe

```text
mb alias unset <nome> [<nome>...] [--mbcli-yaml]
```

- **`--mbcli-yaml`** — remove apenas entradas em `mbcli.yaml`; não altera `aliases.yaml` nem scripts de shell.
- **Validação antes de tudo:** cada nome é checado **antes** de qualquer pergunta ou gravação. Se algum não existir: sem `--mbcli-yaml`, o erro começa por `aliases inexistentes: ...`; com `--mbcli-yaml`, por `aliases inexistentes em mbcli.yaml: ...` (e uma dica para usar `mb alias unset` sem a flag se quiser remover do `aliases.yaml`). Em ambos os casos **nada é alterado**.
- **Nomes repetidos** na linha de comando são tratados uma vez só, na ordem da **primeira** aparição (ex.: `a b a` vira `a b`).
- **Uma confirmação para o lote inteiro.** Se você cancelar, **nenhum** dos nomes é removido.
- **`--yes` / `-y`** — pula a pergunta para **todos** os nomes da lista (no Cobra, a flag pode ir no fim: `mb alias unset a b --yes`).

```bash
mb alias unset dev
mb alias unset api worker
mb alias unset api worker --yes

mb alias unset ci --mbcli-yaml --yes
```
