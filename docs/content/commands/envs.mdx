### `mb envs set`

Define ou atualiza variĂˇveis.

```bash

mb envs set A=1 B=2 C=3 --vault staging

mb envs set A=1 B=2 C=3 --secret      # mĂşltiplas chaves de uma vez
```

Com **`--secret`** ou **`--secret-op`**, pode omitir o valor na linha de comandos: indique sĂł a chave (sem `=`) e o MB pede o valor com **`gum input --password`** (um prompt por chave, em sequĂŞncia), para o segredo nĂŁo ir para o histĂłrico da shell. Se `gum` nĂŁo existir no `PATH`, usa-se leitura mascarada em `/dev/tty`. Em modo nĂŁo interactivo (sem TTY), use sempre `KEY=VALOR` na linha de comandos.

Se usar **`KEY=VALOR`** com `--secret` / `--secret-op`, o MB emite um **aviso** (o valor na linha de comandos pode ficar no histĂłrico da shell e em registos de processos); o fluxo recomendado continua a ser sĂł a chave + prompt mascarado.

```bash

mb envs set A B C --secret                # trĂŞs prompts em sequĂŞncia
mb envs set TOKEN=abc --secret-op         # sem prompt (valor jĂˇ veio na linha)
```

| Flag | DescriĂ§ĂŁo |
|---|---|
| `--vault <nome>` | Grava em `.env.<nome>` em vez do padrĂŁo. NĂŁo aceita `project` nem prefixo `project/` (reservados) |
| `--secret` | Guarda no keyring do sistema (lista `.secrets`); chave sem `=` pede valor com `gum input --password` |
| `--secret-op` | Guarda no 1Password (referĂŞncia em `*.opsecrets`, requer `op`); chave sem `=` pede valor com `gum input --password` |
| `--yes` | Confirma sem prompt para `--secret-op` (CI) |

> `--secret` e `--secret-op` sĂŁo mutuamente exclusivos. Sem flags de store, a variĂˇvel `MB_ENVS_SECRET_STORE` (`plain` / `keyring` / `op`) no ambiente define o destino.

## IntegraĂ§ĂŁo com 1Password (`--secret-op`) {/* #envs-1password-secret-op */}
